# 热血传奇 Go 重制版 — 客户端 & 服务端同步开发计划

## 设计理念

### 同步开发策略

本计划采用**客户端与服务端同步开发**的方式，每个阶段同时推进两端的工作。每个阶段结束时都有一个**可运行的集成里程碑**，确保两端始终保持兼容。

```
Phase 1: 基础层（共享协议 & 类型）
    ↓
Phase 2: 客户端骨架 + 服务端骨架（并行）
    ↓
Phase 3: 渲染层 + 世界层（并行）→ 里程碑：地图可视化
    ↓
Phase 4: 角色层 + 游戏逻辑层（并行）→ 里程碑：角色可见可移动
    ↓
Phase 5: 网络层 + 消息处理层（并行）→ 里程碑：端到端登录
    ↓
Phase 6: 战斗层 + 魔法层（并行）→ 里程碑：可战斗
    ↓
Phase 7: 社交层 + NPC层（并行）→ 里程碑：可交互
    ↓
Phase 8: 进阶系统 + 打磨（并行）→ 里程碑：生产就绪
```

### 现代架构设计原则

1. **接口驱动设计** — 核心系统通过接口交互，便于测试和替换
2. **消息总线架构** — 客户端和服务端内部使用 channel-based 消息总线
3. **组件化实体系统 (ECS)** — 使用组合模式而非深度继承（优于 Delphi 的 TBaseObject 继承链）
4. **依赖注入** — 核心服务通过构造函数注入，便于单元测试
5. **优雅关闭** — 所有 goroutine 通过 context 和 channel 协调关闭

## Phase 1: 基础层 — 共享数据 & 协议

**目标**: 将 Delphi Common/ 目录的所有类型和6Bit协议移植为可复用的Go包

**复杂度**: M | **预计**: 1-2周

### 任务

#### 1.1 internal/protocol/edcode.go — 6Bit 编解码

移植 Common/EDcode.pas：

- Encode6BitBuf(src []byte) []byte — 3字节→4字符，偏移0x3C
- Decode6BitBuf(src []byte) []byte — 反向
- EncodeMessage(msg DefaultMessage) string / DecodeMessage(s string) DefaultMessage
- EncodeString(s string) string / DecodeString(s string) string
- EncodeBuffer(buf []byte) string / DecodeBuffer(s string, buf []byte)
- 可选 XOR 掩码表 (EncodeBitMasks[256], DecodeBitMasks[256])

#### 1.2 internal/protocol/message.go — 消息类型

移植 Common/Grobal2.pas 消息定义：

- DefaultMessage 结构体: Recog int32, Ident uint16, Param uint16, Tag uint16, Series uint16（16字节，packed）
- MakeDefaultMsg(ident uint16, recog int32, param, tag, series uint16) DefaultMessage
- 所有 CM_* 常量（客户端→服务端，约50个ID）
- 所有 SM_* 常量（服务端→客户端，约100个ID）
- 消息帧格式: '#' + code(1位数字) + 编码数据 + '!'

#### 1.3 internal/protocol/types.go — 共享数据结构

移植 Common/Grobal2.pas 类型：

- 方向常量: DR_UP(0) ~ DR_UPLEFT(7)
- 网格常量: UNITX=48, UNITY=32, LOGICALMAPUNIT=40
- 装备槽常量: U_DRESS(0) ~ U_BUJUK(9)（10个槽位）
- StdItem 结构体（60字节）
- UserItem 结构体（24字节）
- Ability 结构体（50字节）
- Magic 结构体
- MsgHeader 结构体
- 毒类型、攻击模式、物品类型等常量

#### 1.4 internal/protocol/types_test.go — 编码测试

- 往返测试: EncodeMessage → DecodeMessage
- 往返测试: EncodeBuffer → DecodeBuffer（各种大小）
- 验证结构体大小

### 交付物

protocol 包能编解码所有 MIR2 网络消息。通过基于已知 Delphi 编码字节序列的单元测试验证。

#### 2A.3 internal/engine/shader.go — 着色器程序

#### 2A.4 internal/engine/resourcemanager.go — WIL 资源缓存

#### 2A.5 internal/engine/scene.go — 场景状态机

## 关键设计决策

1. **internal/protocol/ 集中共享类型** — 客户端和服务端从同一包导入，保证线路兼容性。

2. **internal/engine/ 共享渲染** — 从mapviewer/wilviewer提取OpenGL样板代码到共享包。

3. **internal/game/ ECS 架构** — 使用组件化实体系统替代深度继承，更灵活、更易测试。

4. **internal/storage/ 数据层** — 统一封装 SQLite 操作，便于维护和测试。

5. **单端口统一监听** — server 监听一个端口（默认7000），内部自动路由登录/角色/游戏消息。

6. **每连接一个goroutine** — 每个客户端连接有自己的goroutine。通过channel向游戏引擎线程传递消息。

7. **单线程游戏循环** — 10 ticks/秒在单个goroutine中运行。避免游戏状态的并发修改。

8. **SQLite 数据存储** — 账号、角色、行会数据统一存储在 SQLite 数据库中。

9. **接口驱动设计** — 核心系统通过接口交互，便于单元测试和替换实现。

10. **Channel-based 消息总线** — 客户端和服务端内部使用 channel 传递消息，避免全局状态。

11. **客户端动作锁定** — 必须等待服务器确认后才能发送下一个动作。

12. **6Bit编码保真** — 精确重新实现Delphi编码，不做重新设计。

13. **不使用外部游戏引擎** — 自定义OpenGL渲染，保持依赖最小化。
