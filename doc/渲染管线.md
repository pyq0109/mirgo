# 渲染管线

本文档描述 mirgo 项目中从 WIL 资源加载到屏幕显示的完整渲染管线，涵盖三个可执行程序（cmd/client、cmd/mapviewer、cmd/wilviewer）共用的基础设施。

## 1. WIL 图像加载

### 1.1 文件格式

WIL 文件是热血传奇的图像资源容器，存储像素数据的有序集合。每个图像由 `ImageInfo` 头描述位置和尺寸，像素数据紧跟其后。

**两种像素格式：**
- **8 位调色板**（colorCount ≤ 256）：每个像素 1 字节索引，查调色板表得 RGBA
- **16 位 RGB565**（colorCount > 256）：每个像素 2 字节，直接解码为 RGB

### 1.2 扫描线方向

WIL 文件格式存储像素时采用**底到顶**（bottom-to-top）扫描线顺序，与 BMP 格式相同。即文件中第一个像素对应图像的**左下角**。

`internal/wil/wil.go` 加载后会执行行翻转，将底到顶的扫描线转为 Go `image.RGBA` 标准的**顶到底**顺序：

```go
// WIL stores scanlines bottom-to-top; flip to top-to-bottom for image.RGBA.
stride := w * 4
tmp := make([]byte, stride)
for r := 0; r < h/2; r++ {
    top := rgba.Pix[r*stride : (r+1)*stride]
    bot := rgba.Pix[(h-1-r)*stride : (h-r)*stride]
    copy(tmp, top)
    copy(top, bot)
    copy(bot, tmp)
}
```

翻转后 `image.RGBA.Pix[0:stride]` 是图像的**顶行**，符合 Go 标准约定。

## 2. OpenGL 纹理上传

### 2.1 纹理坐标约定

OpenGL `glTexImage2D` 的规约：像素数据的**第一行**对应纹理坐标 **V=0**（纹理底部）。

对于翻转后的 WIL 图像：
- `Pix[0:stride]`（顶行）→ 上传后位于纹理底部（V=0）
- `Pix[末行]`（底行）→ 上传后位于纹理顶部（V=1）

### 2.2 UploadTexture

`internal/engine/gl.go` 中的 `UploadTexture` 直接上传 `image.RGBA.Pix` 到 GPU，不做额外翻转：

```go
func (s *GLState) UploadTexture(img *image.RGBA) uint32 {
    // ...
    gl.TexImage2D(gl.TEXTURE_2D, 0, gl.RGBA,
        int32(img.Bounds().Dx()), int32(img.Bounds().Dy()),
        0, gl.RGBA, gl.UNSIGNED_BYTE, unsafe.Pointer(&img.Pix[0]))
    // ...
}
```

mapviewer 和 wilviewer 各自有独立的 `UploadTexture` 实现，逻辑相同。

## 3. 渲染几何

### 3.1 Unit Quad VBO

所有纹理绘制共用一个单位四边形 VBO（`internal/engine/gl.go`）：

```go
vertices := []float32{
    // posX, posY,  U, V
    0, 0,  0, 0,  // 左上
    1, 0,  1, 0,  // 右上
    1, 1,  1, 1,  // 右下
    0, 0,  0, 0,  // 左上
    1, 1,  1, 1,  // 右下
    0, 1,  0, 1,  // 左下
}
```

UV 坐标与顶点位置一一对应：左上角 UV=(0,0)，右下角 UV=(1,1)。

### 3.2 Model Matrix

`DrawQuad(x, y, w, h)` 通过 model matrix 将单位四边形变换到屏幕位置：

```
model = translate(x, y) × scale(w, h)
```

变换后：顶点 (0,0) → 屏幕 (x, y)，顶点 (1,1) → 屏幕 (x+w, y+h)。

## 4. 投影矩阵

### 4.1 OrthoProj（Y-down 正交投影）

客户端使用 `OrthoProj(width, height)` 生成 Y-down 正交投影：

```
| 2/w    0     0   0 |
| 0    -2/h    0   0 |
| 0      0    -1   0 |
| -1     1     0   1 |
```

- Y 轴取反（`-2/h`），使屏幕坐标 Y-down：Y=0 是屏幕顶部，Y=height 是屏幕底部
- 屏幕左上角 (0,0) → NDC (+1,+1)（OpenGL 视口左上角）
- 屏幕右下角 (w,h) → NDC (-1,-1)（OpenGL 视口右下角）

### 4.2 OrthoProj4（通用正交投影）

mapviewer 和 scene_play 使用 `OrthoProj4(left, right, bottom, top)`：

```
| 2/(r-l)      0         0   0 |
|    0      2/(t-b)      0   0 |
|    0         0        -1   0 |
| -(r+l)/(r-l) -(t+b)/(t-b) 0 1 |
```

## 5. 纹理方向修正

### 5.1 问题

WIL 图像翻转后（顶到底），上传到 OpenGL 纹理后第一行（顶行）位于 V=0（纹理底部）。在 Y-down 投影下：
- 屏幕顶部（pos.y=0）→ UV V=0 → 采样纹理底部 → 实际是图像顶行 → **正确**
- 屏幕底部（pos.y=1）→ UV V=1 → 采样纹理顶部 → 实际是图像底行 → **正确**

因此客户端（cmd/client）的纹理方向在 WIL 行翻转后是正确的。

### 5.2 mapviewer 的 shader flipV

mapviewer 的 `DrawQuad` 支持 `flipV` 参数，在 fragment shader 中翻转 V 坐标：

```glsl
if (u_flip_v) uv.y = 1.0 - uv.y;
```

WIL 行翻转前，mapviewer 需要 `flipV=true` 来补偿底到顶的扫描线。行翻转后，WIL 图像已是顶到底，不再需要 shader 翻转，因此 `flipV` 设为 `false`。

### 5.3 wilviewer 的 ImGui UV 翻转

wilviewer 通过 ImGui 渲染 WIL 纹理。ImGui 的 OpenGL 后端直接传递 UV 坐标到 fragment shader。

WIL 行翻转前，wilviewer 使用 UV(0,1)→(1,0) 来翻转纹理显示。行翻转后改为标准 UV(0,0)→(1,1)。

## 6. 文字渲染

### 6.1 字形光栅化

`internal/engine/text.go` 的 `TextRenderer` 使用 `golang.org/x/image/font/opentype` 光栅化 TTF 字形到 `image.RGBA`。Go 的 `font.Drawer` 按**顶到底**顺序写入像素（与 `image.RGBA` 约定一致）。

字形纹理通过 `UploadTexture` 上传，方向与翻转后的 WIL 纹理一致，因此共用同一 VBO 无需特殊处理。

### 6.2 字形定位

```go
dx := cursorX + float32(glyph.bearingX)     // 左侧 bearing 偏移
dy := y + float32(tr.ascent - glyph.bearingY) // 基线到字形顶部的距离
```

`ascent` 是字体的上升高度（基线到行顶），`bearingY` 是字形顶部到基线的距离。

## 7. 三条渲染路径

### 7.1 cmd/client — 纯 OpenGL

```
WIL 加载 (行翻转) → UploadTexture → DrawQuad(VBO) → OrthoProj(Y-down)
```

所有场景（登录、选角、游戏、角色动画、文字）共用同一 VBO 和投影矩阵。

### 7.2 cmd/mapviewer — OpenGL + ImGui

```
WIL 加载 (行翻转) → UploadTexture → DrawQuad(flipV=false) + ImGui(小地图)
```

地图瓦片用 OpenGL 渲染，小地图和 UI 用 ImGui 渲染。

### 7.3 cmd/wilviewer — ImGui 为主

```
WIL 加载 (行翻行) → UploadTexture → ImGui.Image(UV(0,0)→(1,1))
```

缩略图网格、预览面板均通过 ImGui 渲染。OpenGL `DrawQuad(flipV)` 路径存在但未使用（死代码）。

## 8. 关键约束

- **WIL 行翻转是全局的**：`internal/wil/wil.go` 的翻转影响所有消费者。修改翻转逻辑需同步检查三个可执行程序。
- **VBO 是全局的**：`internal/engine/gl.go` 的 unit quad VBO 被客户端所有场景共用。修改 UV 坐标需同步检查文字渲染。
- **各工具有独立的翻转逻辑**：mapviewer 的 shader `u_flip_v` 和 wilviewer 的 ImGui UV 翻转已与 WIL 行翻转对齐。新增渲染路径时不要引入额外翻转。
